面白いのは、3件が「バラバラの相手」ではあっても**バラバラのトピックではない**ってところ。3件とも全部レビュー／承認フローの話だよね:

- Sam → レビューの折り返しが遅い(**スループット**)
- Priya → 誰が担当として妥当か(**アサインのルール**)
- Wei → hotfix が止まる(**緊急時の詰まり**)

つまり相手は3人だけど、詰まってる場所は1箇所。**「Kim 対 チーム(人の問題)」に見えるけど、実態は「チームの壊れたレビュー動線が、たまたま Kim の座ってるチョークポイントで発火してる(仕組みの問題)」**という可能性が高い。

## 何が起きてると思うか

Kim がレビュー動線のボトルネック(必須レビュアー / 唯一の承認者 / そのエリアのオーナー)になっていて、負荷を全部一人で受け止めてるんだと思う。そうなると:

- 捌ききれず折り返しが遅れる → Sam と揉める
- 「自分ばかりに来る」ので誰が見るべきか押し返す → Priya と揉める
- 通常レビューの列に hotfix まで並んで詰まる → Wei と揉める

**共通項が Kim だから Kim が原因、という見方は attribution error になりやすい**。人の性格は一週間では変わらない。**一週間で急に発火した=最近“構造の変化”があった**サインなんだよね。誰かが抜けて Kim が単独レビュアーになった / 新しいレビュー方針が入った / hotfix の量が急増した、みたいな。そこを疑いたい。

もう一つ現実的な対抗仮説として「Kim 自身のバーンアウトで当たりがきつくなってる」もある。ただこれも“原因”というより、チョークポイントに置かれ続けた**結果**である公算が大きい。どっちにせよ次の一手は同じ方向を向く。

## どう動くべきか

**まず検証(仮説を先に確かめる、動くのはその後)**
- この3件が本当に全部 Kim を経由してるか、レビュー割り当て・承認ログで確認する
- 同じ摩擦を他のメンバーも踏んでないか(Kim だけの問題か、動線の問題か)を見る

**Kim との1on1 — フレーミングを間違えない**
- 「最近みんなと衝突してるね」ではなく「**この動線、君を十字砲火の真ん中に置いちゃってる**。負荷どうなってる?」から入る。Kim を犯人にした瞬間に本当に孤立する
- 実際のレビュー本数・待ち行列・割り込み頻度を数字で聞く

**仕組みを直す(3件を1つずつ仲裁しない)**
- Sam の件 → レビュー担当のローテ/負荷分散、折り返しの目安(SLA)を明文化
- Priya の件 → 「誰が見るか」を毎回交渉させない割り当てルール(オーナー表 / round-robin)
- Wei の件 → hotfix が通常レビュー列に並ばない**緊急パス**(専任 or ペアレビューで即承認)

要は、3つの人間関係トラブルとして個別に消火するんじゃなく、**「Kim を単一障害点にしているレビュー動線」を1つの問題として直す**。そうすれば Kim の孤立も自然に解ける。もし動線を直しても摩擦が残るなら、そのとき初めて Kim 個人の要因(疲弊・コミュニケーション)にフォーカスすればいい。順番が逆になると、直せる仕組みの問題を人のせいにして Kim を本当に追い出すことになる。

まず「3件が全部 Kim 経由か」を確認するところから始めるのがいいと思う。そこが Yes なら、ほぼ仕組みの話で確定。
