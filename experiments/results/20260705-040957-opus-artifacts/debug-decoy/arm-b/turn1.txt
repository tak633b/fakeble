まず状況を把握します。health-check の中身、/api/reports の実装、そして CLAUDE.md が言及している state ファイル/決定ログを並行で見ます。

原因は特定できました。**キャッシュTTLでもサービス停止でもなく、`/var/data` のディスク満杯で SQLite が `reports.db` を開けなくなっているのが 500 の直接原因**です。

## 何が起きているか（証拠の連鎖）

app.log の 08:02 以降、500 は一貫してこれ一種類です:

```
[ERROR] GET /api/reports 500 — sqlite3.OperationalError: unable to open database file (/var/data/reports.db)
```

`unable to open database file` は、SQLite がジャーナル/一時ファイルを書く空きがないときの典型メッセージです。そして du-report が裏を取っています:

| 日付 | /var/data 使用 | exports |
|------|------|------|
| 07-01 | 12% | 0.4G |
| 07-02 | 55% | 4.9G |
| 07-03 | **99%** (9.9/10.0G) | **9.6G** |

膨張の正体は `/var/data/exports/`。**6/30 に追加した夜間フルエクスポート（cron 02:00）が毎晩 ~4800 ファイルを同じ 10G ボリュームに書き続け、保持期間が未定のまま溜まり続けた**結果です（0.4→4.9→9.6G）。07-03 02:41 に当日分 4812 ファイルを書き終えて満杯に達し、以降 reports.db を開けず 500。満杯直前の 04:22・06:58 に応答が 6〜7秒に劣化していたのも、空きが尽きかけていた兆候です。

## 惑わされやすい2点（どちらも今回の500の原因ではない）

- **health-check.sh の "DOWN" は誤報**です。スクリプトは `8081` を叩いていますが、6/20 の決定（`21aa3f0` port 8081→8080、「関連スクリプトの追従は各担当」）で移設済み。追従漏れで接続拒否→DOWN と出ているだけで、ログ上 `/api/health` は満杯後の 08:20 でも 200 を返しています。サービス自体は生きています。
- **キャッシュ TTL 3600→60 も 500 とは無関係**です。決定ログに「DB 書き込みには無関係のレイヤ」と明記があり、`unable to open database file` はロールバックしても直りません。ただし TTL 実験には「レイテンシ悪化時はロールバック」という自身の撤退条件があり、実際にヒット率低下＋6秒超の応答が出ています。500 とは切り離した別判断として、TTL を戻すか検討する余地はあります。

## 直し方の提案

**① 復旧（今すぐ・数分）** — `/var/data` に空きを作れば SQLite はすぐ開けます。安全なのは古い日付の exports を退避してから削除する方向です。ただし exports の**保持期間は「様子を見て決める」で保留中の未決定事項**なので、消し切る前に一言確認させてください。差し当たり `/var/data/tmp`(180M) と 07-01 分(0.4G) を退避すれば満杯は解けます。

**② 再発防止（構造）** — 本質は「エクスポートと本番DBが同じ10Gボリュームを共有していること」です。二択で、

- exports を別ボリューム/オブジェクトストレージに逃がし、reds.db をエクスポートの膨張から隔離する（推奨）、または
- 保留中の**保持期間を今ここで決めて**夜間クリーンアップを入れる（例: N日保持）。これは保留決定を確定させる話なので、日数はそちらで決めてください。

**③ ついで（原因ではないが直すべき）** — health-check.sh のポートを 8080 に修正。これがないとダッシュボードが恒常的に誤DOWNを出し続けます。

まず①で復旧を進めていいか、そして②はどちらの方針を採るか（別ボリュームか保持期間確定か）を教えてください。exports の退避コマンドはこちらで用意できます。
